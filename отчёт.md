# Отчёт по Модулю А — Сбор и предварительная обработка данных

## 1. Загрузка данных

Реализован модуль `dataLoader.py`, выполняющий роль точки входа системы. Функция `main(path_to_data)` принимает путь к директории с учебными материалами и итеративно обходит все вложенные папки. Для каждой директории создаётся объект класса `Analizer` (`analizer.py`), который извлекает содержимое файлов.

```python
# dataLoader.py
def main(path_to_data:str):
    analizer = Analizer(client, "sage-mm-qwen3-vl-4b-sft")
    for idx, folder in enumerate(os.listdir(path_to_data)):
        text = analizer.getText(os.path.join(path_to_data, folder))
        ...
```

Система запускается командой `python dataLoader.py` с заданной корневой директорией `"data"`.

После извлечения текста для каждого учебного материала вызывается функция `get_annotation()` из модуля `annotation.py`. Она обращается к языковой модели и возвращает структурированный JSON со следующими полями:

| Поле | Описание |
|---|---|
| `subject` | Предмет (дисциплина): одно из шести значений (Машинное обучение, Математика и др.) |
| `topic` | Тема материала |
| `annotation` | Краткая аннотация |
| `difficulty_level` | Уровень сложности: `easy`, `medium`, `hard` |
| `type` | Тип учебного материала |

Каждая запись также получает `id` (порядковый номер) и полный извлечённый `text`. Все поля сохраняются в строку датафрейма `pandas`:

```python
rows.append({
    "id": idx,
    "text": text,
    "annotation": annotation["annotation"],
    "subject": annotation["subject"],
    "topic": annotation["topic"],
    ...
})
```

Для формирования аннотации применяется строгая JSON-схема (`response_format`), что гарантирует структурированный и предсказуемый вывод модели. Запросы к LLM кэшируются механизмом `@persistent_cache`, что исключает повторную обработку одних и тех же материалов.

## 2. Соответствие методическим рекомендациям

Реализован модуль `recomendations.py` с функцией `get_recomendation(text, client)`. Агент читает файл методических рекомендаций (`recomendations.md`) и анализирует каждый учебный материал на соответствие им. Результат — числовые оценки и текстовый вывод:

| Поле | Описание | Диапазон |
|---|---|---|
| `struct` | Структура материала | 0.0 – 1.0 |
| `validity` | Методическая обоснованность | 0.0 – 1.0 |
| `availability` | Доступность и язык изложения | 0.0 – 1.0 |
| `summary` | Краткий вывод о соответствии (до 250 символов) | — |

Поля `rec_struct`, `rec_validity`, `rec_availability`, `rec_sum` сохраняются в итоговый набор данных для каждого материала. Каждая аналитическая запись содержит явный вывод о допустимости использования материала.

## 3. Хранение данных и обновление без дублей

После формирования датафрейма вызывается функция `save_df_to_postgres(df, "data", POSTGRES)` из модуля `storage.py`, которая сохраняет все записи в таблицу `data` базы данных PostgreSQL. Параллельно данные сохраняются в локальный CSV-файл `orig.csv` для резервного доступа.

Исключение дублей обеспечивается двумя механизмами:
1. **Кэширование запросов** (`@persistent_cache`) — повторный вызов с теми же параметрами не приводит к новому обращению к LLM и не создаёт новую запись.
2. **Идентификатор** — каждый материал получает уникальный `id` на основе порядкового номера папки; при переносе в БД используется этот ключ.

Повторный запуск `dataLoader.py` заново обходит директорию `data`. Если файл не изменился — результат берётся из кэша, запись не дублируется. Если появился новый материал — он обрабатывается и добавляется. Модуль `eduGenerator.py` реализует аналогичную логику для генерируемых материалов, явно отделяя их флагом `"generated": True`.

## 4. Формирование набора данных

### Поддерживаемые форматы файлов

Метод `getText()` класса `Analizer` поддерживает следующие форматы:

| Расширение | Метод извлечения |
|---|---|
| `.txt`, `.md` | Прямое чтение с кодировкой UTF-8 |
| `.pdf` | Библиотека `pypdf` (постраничное извлечение) |
| `.doc`, `.docx` | Библиотека `python-docx` с сохранением порядка параграфов и таблиц |
| `.ipynb` | Парсинг JSON, извлечение содержимого ячеек (`source`) |
| `.pptx` | Библиотека `python-pptx`, извлечение текста с каждого слайда |

Обход директорий выполняется рекурсивно через `Path.rglob("*")`, что охватывает вложенные папки любой глубины. Имя файла добавляется к тексту в качестве заголовка для трассируемости.

### Обработка медиаконтента

Метод `getMedia()` класса `Analizer` обрабатывает нетекстовые файлы:

- **Изображения** (`.jpeg`, `.png`, `.webp`) — передаются в функцию `get_image_annotation()` из модуля `image.py`, которая использует мультимодальную языковую модель для получения текстового описания.
- **Видео** (`.mp4`, `.avi`, `.mov`, `.mkv`, `.wmv`, `.webm`) — обрабатываются функцией `get_video_annotation()` из модуля `video.py`.

Результат сохраняется в поле `media_annotation` каждой записи набора данных.

### Структура итогового набора данных

Итоговый датафрейм, сохраняемый в PostgreSQL (таблица `data`) и в файл `orig.csv`, содержит следующие поля:

| Поле | Тип | Описание |
|---|---|---|
| `id` | int | Идентификационный номер |
| `subject` | str | Предмет (дисциплина) |
| `topic` | str | Тема материала |
| `text` | str | Полный текстовый материал |
| `annotation` | str | Краткая аннотация |
| `media_annotation` | str | Описание медиаконтента |
| `rec_struct` | float | Оценка структуры (0–1) |
| `rec_validity` | float | Оценка методической обоснованности (0–1) |
| `rec_availability` | float | Оценка доступности изложения (0–1) |
| `rec_sum` | str | Итоговое заключение о допустимости использования |
| `generated` | bool | Признак автоматической генерации |
| `type` | str | Тип учебного материала |
| `difficulty_level` | str | Уровень сложности |
| `previous_id` | int | ID предыдущего материала в цепочке изучения |

## 5. Предобработка данных и выделение признаков

Анализ атрибутов проводится на уровне аннотирования: языковая модель извлекает семантически значимые признаки — предмет, тему и уровень сложности. Эти признаки используются далее как основа для сравнения материалов при построении порядка изучения и определении пропущенных тем. Схожесть материалов оценивается агентом `gen_check.py` через сопоставление полей `subject` и `annotation` по всем записям одной дисциплины.

Реализован модуль `order.py` с функцией `get_order(df, client)`. Агент получает список всех материалов с их `id`, `subject` и `annotation`, после чего определяет логическую последовательность изучения: для каждого материала указывается `previous_id` — идентификатор материала, который необходимо изучить перед текущим и `next_id` — идентификатор материала, который можно изучить после текущим

## 6. Статистический анализ набора данных

После формирования основного датафрейма вычисляются производные статистические поля:

```python
df['text_len'] = df['text'].str.len()
df["level_from_midle"] = df["text_len"] - df["text_len"].mean()
save_df_to_postgres(df, "stats_data", POSTGRES)
```

На основе поля `text` вычисляются:
- **Длина текста** (`text_len`) — количество символов в материале.
- **Отклонение от среднего** (`level_from_midle`) — позволяет выявить материалы, значительно превышающие или уступающие среднему объёму.

Таблица `stats_data` в PostgreSQL содержит полный набор статистических данных, на основе которых дашборд строит аналитику по частоте значений предметов, типов и уровней сложности.

## 7. Расширение набора данных

Модуль `gen_check.py`, функция `gen_if(df, client)`, анализирует имеющиеся материалы по каждой дисциплине и формирует список тем, которые необходимо добавить. Агент использует поля `subject` и `annotation` для оценки покрытия. Пользователю предлагается интерактивное подтверждение:

```python
if input(f"для {item['subject']} не хватает тем {result['items']}. 
          Хотите сгенерировать учебные материалы на эти темы?(y/n)") != 'y':
    return None
```

При подтверждении агент последовательно генерирует учебные тексты по каждой из предложенных тем на материал:

```python
response = client.chat.completions.create(
    model=TEXT_MODEL,
    messages=[
        {"role": "system", "content": "ты агент для написания учебных материалов..."},
        {"role": "user", "content": f"напиши учебный материал на тему: {topic}"}
    ],
    temperature=0.7,
    max_tokens=4000
)
```

После генерации каждый новый материал автоматически проходит полный цикл обработки — те же шаги, что и для загруженных материалов:
1. Получение аннотации через `get_annotation()`.
2. Проверка на соответствие методическим рекомендациям через `get_recomendation()`.
3. Присвоение флага `"generated": True` для отличия от исходных материалов.
4. Сохранение в PostgreSQL через `save_df_to_postgres()`.

Этот процесс реализован в функции `generate_if_need()` как в `dataLoader.py`, так и в `eduGenerator.py`.
